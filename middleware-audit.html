1. Code Duplication & Structural Redundancy 4 findings

F1 — defineTaskMiddleware / defineResourceMiddleware are near-identical
medium risk
src/definers/defineTaskMiddleware.ts (L1–162) · src/definers/defineResourceMiddleware.ts (L1–162)
redundant

What: Both files are 162 lines, structurally identical. They differ only in which symbol is used (symbolTaskMiddleware vs symbolResourceMiddleware), the type imports, and the assertTagTargetsApplicableTo kind string. The wrap() closure, resolveCurrent() logic, .with() implementation, config merging, freezing — all copy-pasted.

Why simplify: A single generic defineMiddleware(kind, def) parametrized by symbol/type would eliminate ~160 lines of duplication. Any bug fix or feature (e.g. new .with() behavior) must currently be applied in both files.

F2 — TaskMiddlewareComposer / ResourceMiddlewareComposer parallel structures
medium risk
src/models/middleware/TaskMiddlewareComposer.ts (417 lines) · src/models/middleware/ResourceMiddlewareComposer.ts (282 lines)
redundant

What: Both composers follow the identical pattern: createBaseRunner → applyLocalInterceptors → applyMiddlewares → applyGlobalInterceptors → wrapWithInterceptors. The wrapWithInterceptors, applyGlobalInterceptors, and applyMiddlewares methods are structurally the same, only differing by whether they operate on ITaskMiddlewareExecutionInput (with journal + source) or IResourceMiddlewareExecutionInput (without).

Why simplify: ~700 combined lines could likely be reduced to ~350 with a generic composition utility. The resource composer even duplicates the entire wrapWithInterceptors onion logic.

F3 — timeout task and resource middleware duplicate the entire Promise-race pattern
low risk
src/globals/middleware/timeout.middleware.ts — L56–106 (task) vs L110–164 (resource)
redundant

What: The task timeout implementation (50 lines of Promise/settle/abort logic) is copy-pasted almost verbatim for the resource variant. The only meaningful difference is the task version reads from an existing journal abortController while the resource version always creates one fresh.

Why simplify: Extract a shared withTimeout(ttl, fn) utility. The settle/clearTimeout/abort pattern is an implementation detail that should live in one place.

F4 — Parallel ITaskMiddleware / IResourceMiddleware type hierarchies
high risk
src/types/taskMiddleware.ts (118 lines) · src/types/resourceMiddleware.ts (107 lines)
redundant complexity

What: These two files define nearly mirror-image type hierarchies: ITaskMiddlewareDefinition / IResourceMiddlewareDefinition, ITaskMiddleware / IResourceMiddleware, ITaskMiddlewareConfigured / IResourceMiddlewareConfigured, ITaskMiddlewareExecutionInput / IResourceMiddlewareExecutionInput. The only structural difference is that task execution input includes journal while resource does not.

Why simplify: A generic IMiddleware<Kind, TConfig, TInput, TOutput, TDeps> base type with kind-specific execution input would halve the type surface. This is the deepest change — it affects every consumer — hence high risk, but it's the root cause of findings F1–F3.

2. Built-in Middleware Overlap & Consolidation 3 findings

F5 — retry + timeout coupling via journal keys
low risk
src/globals/middleware/retry.middleware.ts — L5 (import of timeoutJournalKeys), L69–73, L82–85
overlap

What: The retry middleware directly imports and reads timeoutJournalKeys.abortController to decide whether to stop retrying after a timeout abort. This creates a hard coupling between two independently-configured middleware. Retry reaches into timeout's journal state twice — once to check signal.aborted, once to pass the signal to abortableDelay().

Why simplify: Instead of retry knowing about timeout's internals, the abort controller could be a first-class execution context property (on the journal by convention or on the execution input). Or retry could simply check whether the caught error is a TimeoutError instance — which it already could do without the journal coupling.

F6 — circuitBreaker + rateLimit share a similar "reject when threshold exceeded" pattern
low risk
src/globals/middleware/circuitBreaker.middleware.ts (203 lines) · src/globals/middleware/rateLimit.middleware.ts (130 lines)
overlap

What: Both maintain per-task counters in a companion resource, both throw custom RunnerError subclasses when a threshold is hit, both expose journal keys for current state. While their semantics differ (error-count vs request-count), the scaffolding (resource with state map, threshold check, journal sync) is templated.

Why simplify: Not a direct consolidation candidate (distinct policies), but a shared "stateful middleware scaffold" pattern could reduce boilerplate. Low risk because the semantics genuinely differ. Primarily a DX improvement if more such middleware are planned.

F7 — debounce & throttle could share more temporal infrastructure
low risk
src/globals/middleware/temporal.middleware.ts — L112–188 (debounce) vs L194–289 (throttle)
overlap

What: Already partially consolidated under temporalResource, but the two middleware bodies duplicate the pattern of: check disposed → get/create state from WeakMap → setup resolve/reject lists → schedule setTimeout → on fire, drain lists. The rejectDebounceState and rejectThrottleState helpers are near-identical (differ by currentPromise cleanup).

Why simplify: A shared "temporal batcher" abstraction could encapsulate the resolve/reject list draining and disposal rejection. Minor win (~30 lines saved) but improves correctness: the disposal path only needs to be maintained once.

3. Overly Complex Configuration & Patterns 3 findings

F8 — cache middleware config schema exhaustively enumerates all LRUCache options
medium risk
src/globals/middleware/cache.middleware.ts — L85–115 (cacheMiddlewareConfigPattern, 31 individually listed Match.Optional fields)
complexity

What: The config schema manually mirrors every single property of LRUCache.Options as individual Match.Optional assertions: ttl, max, ttlAutopurge, allowStale, updateAgeOnGet, updateAgeOnHas, maxSize, maxEntrySize, ttlResolution, ttlAutopurgeWarn, noDeleteOnFetchRejection, noDeleteOnStaleGet, noDisposeOnSet, fetchMethod, sizeCalculation, dispose, disposeAfter, noUpdateTTL, noDeleteOnStaleFetchRejection, allowStaleOnFetchAbort, allowStaleOnFetchRejection, ignoreFetchAbort, forceRefresh, noDeleteOnFetchAbort, size, stale. This will break whenever LRUCache updates its options.

Why simplify: Replace with Match.ObjectIncluding({ keyBuilder: Match.Optional(Function) }) and pass through the rest unvalidated (LRUCache will validate its own options). Only validate runner-specific options like keyBuilder. Reduces risk of version mismatch with the lru-cache dependency.

F9 — Contract middleware generics (TEnforceInputContract, TEnforceOutputContract)
high risk
src/types/taskMiddleware.ts — L25–39 (4-param generic on ITaskMiddlewareDefinition) · src/types/contracts.ts (219 lines of type-level machinery) · src/definers/defineTaskMiddleware.ts — L22–39 (4-param generic on defineTaskMiddleware)
complexity

What: Every middleware definition/instance/configured type carries <TConfig, TEnforceInputContract, TEnforceOutputContract, TDependencies> — 4 generic parameters. The contract types exist purely for compile-time enforcement via an elaborate 219-line contracts.ts with UnionToIntersection, IsImpossibleIntersection, KeysWithNever, InputContractViolationError, etc. Most user-authored middleware uses void for both contract params (the default).

Why simplify: The input/output contract enforcement on middleware is a powerful type-safety feature for library-level middleware that constrain what tasks they can be attached to. However, the cost is high: every single type signature in the middleware stack (definers, composers, store types, builders) must thread through 4+ generics. Consider whether the contract types could be an opt-in wrapper type rather than baked into every base interface. High risk because this is deeply woven into the type system.

F10 — concurrency middleware config offers 3 overlapping ways to provide a semaphore
medium risk
src/globals/middleware/concurrency.middleware.ts — L11–32 (ConcurrencyMiddlewareConfig), L46–75 (assertConcurrencyConfig)
complexity

What: Config accepts { limit }, { limit, key }, OR { semaphore } — but mixing them is an error (validated at runtime with 3 separate validation branches). The key-based lookup adds another dimension: keyed semaphores are stored globally (in semaphoresByKey) and require a limit-match assertion. This is 3 paths through a single config object, with 3 error cases (semaphore+limit, key without limit, neither).

Why simplify: Consider splitting into { limit } (auto-create) and { semaphore } (bring-your-own) — remove key entirely and let users who need shared keyed semaphores create their own. Or use discriminated config variants: { mode: "auto", limit } | { mode: "shared", key, limit } | { mode: "custom", semaphore }.

4. Journal Keys & Interceptor Overhead 2 findings

F11 — Journal keys proliferation across middleware
low risk
retry.middleware.ts (2 keys) · timeout.middleware.ts (1 key) · cache.middleware.ts (1 key) · circuitBreaker.middleware.ts (2 keys) · rateLimit.middleware.ts (3 keys) · fallback.middleware.ts (2 keys) — Total: 11 journal keys
simplify

What: Each middleware exports its own journalKeys object with typed keys. The journal is a flat key-value map using string IDs. Users must import the correct journalKeys from the correct middleware file to read state. The system works but creates a discovery problem: there's no central registry of all available journal keys, and the naming convention (runner.middleware.task.cache.hit vs runner.middleware.retry.attempt) is inconsistent (some include task in the namespace, some don't).

Why simplify: (a) Standardize the naming convention. (b) Consider a single runner.journal barrel export that aggregates all built-in journal keys. (c) The rateLimit middleware writes 3 keys per execution, two of which (resetTime, limit) are config-derived constants — they could be read from config directly rather than journaled.

F12 — Four-level interceptor hierarchy (global task/resource + per-middleware task/resource)
medium risk
src/models/middleware/InterceptorRegistry.ts (205 lines) · src/models/MiddlewareManager.ts — L91–212 (intercept/interceptOwned/interceptMiddleware/interceptMiddlewareOwned — 8 overloaded methods)
complexity

What: MiddlewareManager exposes interceptors at 4 levels: global task, global resource, per-task-middleware, per-resource-middleware. Each level has two API variants: intercept() (public) and interceptOwned()/interceptMiddlewareOwned() (internal, tracks owner resource). That's 8 overloaded method signatures, plus the InterceptorRegistry with 4 separate storage maps, 4 add methods, 4 get methods, and an owner-snapshot extraction system.

Why simplify: The "owned" variants exist purely for tooling/introspection. Consider: (a) collapsing the owned metadata into optional params on the public API instead of separate methods; (b) questioning whether per-middleware interceptors (vs global interceptors) carry their weight — they add significant code but their use cases overlap with middleware ordering or composition.

Recommendation priority: F1 (definer duplication) and F3 (timeout duplication) are the lowest-effort, highest-confidence improvements. F8 (cache config) is a clear maintainability win. F4/F9 (type unification + contract simplification) are the most impactful but carry the highest risk and touchpoint count. F11 (journal key naming) is a non-breaking improvement that improves DX.
