1 · Tag System — Over-Engineering

# Finding Location Why Eliminate / Simplify Risk
1 TAG Contract tags — TEnforceInputContract / TEnforceOutputContract generics on tags
src/types/tag.ts L48–L98
src/definers/defineTag.ts L33–L47
src/types/contracts.ts (entire file, 219 lines)
Tags carry four generic type params (TConfig, TEnforceIn, TEnforceOut, TAllowedTargets). The contract params are only used for compile-time type narrowing of task inputs/outputs. This machinery adds ~220 lines of advanced generic plumbing (contracts.ts) plus IContractable extension on tags, middleware, etc. Most users never use contract tags — they use schema validation. Simplifying to a separate utility type or removing contract enforcement from the tag layer would dramatically cut type complexity. High
2 TAG Scoped tags — .for() restricting tag targets at the type level
src/types/tag.ts L163–L185
src/definers/builders/tag/fluent-builder.ts L59–L82
TAllowedTargets adds a fourth generic to every tag, plus 7 target-specific type aliases (TaskTagType, ResourceTagType, etc. L178–L185). This is nice DX but adds significant type-level noise everywhere — every builder that accepts tags must carry matching generic filters. A runtime-only check (already present in assertTagTargetsApplicableTo) would suffice for most use cases. Medium
3 TAG .startup() method on tags
src/definers/defineTag.ts L138–L159
 tag.startup() wraps a tag as a "before-init" dependency. This is a very niche DI signal — most resources don't need startup tag ordering. It adds a wrapper type (ITagStartupDependency), a unique symbol, and nested .optional(). Could be a standalone utility rather than a method on every tag instance. Low
4 TAG .schema() alias on tag builder
src/definers/builders/tag/fluent-builder.interface.ts L27–L30
.schema() is an alias for .configSchema() on the tag builder. Two names for the same method adds confusion without adding clarity. Pick one. Low
5 ALIAS globalTags.internal duplicates globalTags.system
src/globals/globalTags.ts L20–L21
 Deprecated alias. internal is just system under a different name. Remove at next major. Low

2 · Error System — Bloat & Feature Creep

# Finding Location Why Eliminate / Simplify Risk
6 ERROR .remediation() on every error definition
src/definers/defineError.ts L130–L137
src/definers/builders/error/fluent-builder.interface.ts L37–L41
~95% of the 80+ error definitions use .remediation() with verbose paragraphs that get appended to every error message. This is useful for framework-internal errors, but it bloats bundle size (remediation strings alone are several KB), pollutes stack traces, and is rarely consumed programmatically. Could be opt-in debug-mode-only, or removed from the public builder API and kept internal. Medium
7 ERROR .is(error, partialData) — partial matching overload
src/definers/defineError.ts L195–L207
 The .is() type guard accepts an optional partialData argument for shallow matching (matchesRunnerErrorData). This is a niche inspection pattern — users rarely check which specific data a typed error carries via its guard. A simple .is(error) guard is sufficient; data inspection can be done after narrowing. The partial matcher function is 15 lines of code that serves very few call sites. Low
8 ERROR .httpCode() on every error builder
src/definers/builders/error/fluent-builder.interface.ts L13
src/definers/defineError.ts L24–L37
 httpCode mixes HTTP transport concerns into the error definition layer. Only 6 out of 80+ errors use it (408, 413, 429, 499, 500, 503). This is really an HTTP exposure concern and could live in the HTTP layer (e.g., a mapping from error id → status code) rather than on every error helper. The assertion logic with isValidHttpCode is duplicated in both defineError.ts and the builder. Medium
9 ERROR .serialize() / .parse() on error builder
src/definers/builders/error/fluent-builder.interface.ts L14–L15
src/types/error.ts L14–L15
 serialize and parse fields exist on the error definition interface, and corresponding builder methods exist, but they are never used at runtime — the ErrorHelper class doesn't call them. They appear to be forward-looking stubs for potential error serialization over the wire, but they pollute every builder's API today with no benefit. Low
10 ERROR .create() deprecated alias for .new()
src/definers/defineError.ts L190–L193
 Marked @deprecated. Should be removed at next major version. Low
11 ERROR .schema() alias for .dataSchema() on error builder
src/definers/builders/error/fluent-builder.interface.ts L22–L25
Same pattern as the tag builder — two names for the same thing. Pick one, remove the other. Low

4 · check() / Match System — Redundancy

# Finding Location Why Eliminate / Simplify Risk
17 UTIL check() and Match — entire subsystem
src/tools/check/ (7 files, ~1500 lines)
src/tools/check/matcher/ (6+ files)
src/tools/check/toJsonSchema.ts (520 lines)
src/tools/check/classSchema.ts (201 lines)
 This is a full Meteor check-style runtime validation library embedded in the framework. It includes pattern matching (Match.OneOf, Match.Where, Match.ObjectStrict, etc.), class schema decorators (@Match.Schema(), @Match.Field()), JSON Schema generation (520 lines), and error types. It overlaps heavily with Zod/Valibot/ArkType which the framework already supports via ValidationSchemaInput. Internal usage is modest — serializer, concurrency middleware config, rpc-lanes config, and durable workflow tags use it. This is a large amount of code to maintain for a convenience validation layer. It could be extracted to a separate package or replaced with the users' choice of schema library. High
18 EXPORT 10 Match-related type exports from main entry
src/index.ts L44 via src/public.ts L155–L165
The main entry re-exports CheckSchemaLike, CheckedValue, InferCheckSchema, InferMatchPattern, MatchCompiledSchema, MatchJsonObject, MatchJsonPrimitive, MatchJsonSchema, MatchJsonValue, MatchPattern. These are internal types of the check system and pollute the public API. Most users using Zod or similar never need these. Medium

5 · Public API Surface — Excess Exports

 Marked @deprecated in favor of r.system, r.runner, r.debug. It aggregates events, resources, system, runner, middleware (and a deprecated middlewares alias), tags, and debug into a single object. This entire export should be removed at next major — it's redundant with the r namespace. Medium

 LockableMap is a 50-line internal utility class (Map with a lock flag). It's exported from the main entry point, but it's an implementation detail used internally by the Store. Library users have no reason to import it. Removing from the public API would reduce surface without impact. Medium

Summary of Recommendations

Quick wins (low risk, immediate cleanup)

8 items — Remove deprecated aliases (circularDependenciesError, dependencyCycleError, platformInvariantError, serializerPayloadValidationError, middlewareInterceptUnknownTypeError, globalTags.internal, .create() on errors), remove unused .serialize()/.parse() stubs from error builder, and collapse .schema() aliases on both tag and error builders.

Medium-term simplification (medium risk, v7 planning)

8 items — Move .httpCode() to the HTTP exposure layer, make .remediation() debug-only or internal, remove globals deprecated namespace, unpublish LockableMap, remove createContext/createTestResource compat exports, collapse inputSchemaValidationError/resultSchemaValidationError aliases, reduce Match type exports from main entry.

Structural changes (high risk, careful migration)

3 items — (1) Simplify or decouple contract tags from the tag generics to reduce type complexity across the entire codebase. (2) Evaluate extracting the check()/Match system (~2000+ lines) into a separate package or replacing internal usage with a lighter pattern. (3) Consider removing the scoped tag target types (TaskTagType, etc.) in favor of runtime-only target enforcement. These are deeply woven into the type system and need careful migration planning.

Error definition count — domain-runtime.errors.ts is 800 lines

The single file src/errors/domain-runtime.errors.ts defines 50+ error helpers with verbose formatting and remediation strings. Combined with foundation errors (~180 lines across 7 files) and model-runtime errors (~177 lines), the error system accounts for ~1,200 lines of code. Consider whether some of these could be collapsed into parameterized generic errors (e.g., one "lane binding not found" error parameterized by lane type, instead of separate rpcLaneBindingNotFoundError and eventLaneBindingNotFoundError).
